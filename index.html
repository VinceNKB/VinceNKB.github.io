我的游戏日志(2): 精灵宝可梦 太阳

发表于 2017-02-08 | 分类于 ACG ， Game |

我玩口袋妖怪的时间有点久了，最早记得是小学的时候在电脑上用GBA模拟器玩蓝宝石。那时候还是用电话线拨号上网，一个几M的ROM要下两个小时。下的还是盗版中文，背包里的道具每一个能看懂意思的。就算如此，还是凭借着年轻时对游戏的热情和S/L大法，花了将近一个暑假，打到了四天王。然后不小心手贱把档删了，感觉就像删掉了自己曾经活着的证据一般。还记得当时选的御三家是沼跃鱼，技能好像是水枪，泥浆喷射，地震和冲浪。想当年HM还不能覆盖，TM只能用一次。用冲浪替掉了浊流真的很后悔，所以为什么还会留着水枪呢？想想当时自己真是年轻，属性相克表都没背全，什么战斗技巧都不懂，但还是玩得很快乐。

宝石之后，平台就变成了NDS，当时的模拟器性能实在太烂，所以就没有机会再玩了。倒是金银水晶版玩了很久。印象最深的是在白银山下花了好久练班吉拉，然后用快龙和班吉拉两只横扫四天王。

然后第一次接触正版的口袋妖怪是在我买了新三大之后。新三大还在从日本回来的路上，卡带已经到手上了。当时买的是x版，玩了大概30个小时，感觉最有趣的就是奇迹交换，传了好多家门怪上去，结果换了好多蛋余和稀有怪，当时还不知道什么是魔法师。还有就是通关以后劲敌要和我交换精灵，然后我就用自己等级最高的草主换了个5级水主。。。不过比较无奈的是，正版并没有中文，所以是选了英文硬上的。

然后就是去年11月，日月上市了，自带简体繁体中文。寒假漫漫，所以我剁手了。

这代估计是初代以后销量最好的一版了，而且极有可能超越初代。毕竟刚刚上市不到3个月，销量就已经超越之前的好几个世代。究其原因，首次加入中文功不可没，毕竟有了中国市场的加入。而另一个原因是游戏的种种革新。虽然口袋妖怪每一代都会有一定程度的革新，但日月的革新可以说是大为出彩。首先是取消了道馆，改为诸岛巡礼，代代都是挑战道馆确实有些无聊；作为实力证明的徽章改为了z结晶；而z技能的加入，从影响上看正如前代加入mega一般，使用z技能的姿势以及名字都相当中二；同时HM被取消，转而代之的是坐骑系统。整体上说，这代对新手相当友好，比如刚才说到的骑行系统，剧情上也是根本不需要看攻略，加入了大量的动画时间，剧情恨不得带着你走一遍，所以这作从演出效果上看也是相当赞的，比如异兽降临时的CG，鸣鸣登场时那个霸气。说到画面，这作取消了3D，当时拿到游戏的时候我还找了好久的3D，后来发现真没了，这作画面上做了更多加强，几乎榨干了3DS的机能，所以根本没有余裕留给3D了，其实使用z机能还有2v2的时候已经让画面有些卡顿。

至于剧情方面，实际上口袋妖怪这种游戏对剧情要求并不高，毕竟作为主体玩家的小学生并不能看懂过于深刻的剧情，所以口袋妖怪作为卖点的更多的是交换和对战，明明是个掌机游戏，却鼓励孩子们跑到户外去交朋友，真是完美的子供向。但这作剧情非常清晰流畅，人物也是性格鲜明，不会碰到玩到最后惊呼这货是谁啊的窘境。

而必须一提的是我的老婆莉莉艾。这孩子展现了一个温柔善良而又坚强的女孩子应该是什么样子的。而且这作的感情线也是相当露骨啊，椰蛋树岛那段还有庆典烟花那段，气氛都差不多是表白了，虽然我选的主角也是女的，但是完全感觉不到违和感。这游戏IGN打了9分，有人吐槽说这游戏之所以被扣了1分是因为莉莉艾最后走了。陪伴我们整个一周目的女孩子最后走了，为了帮助自己的母亲几乎是只身前往关东，如何也想不到那个我们在游戏一开始遇到的被几只烈雀吓跑的女孩子已经有了这样的成长。但她确实走了，而把我们留在了二周目。我通完一周目之后立即就去百度，莉莉艾是不是真的不会回来了。记忆中最深的是在和她一起前往异兽洞窟的时候，在洞口边的她会帮我的妖怪治疗，而在那之后对我说：“我这里还有许多厉害上药和pp单补，但是黄金喷雾的存量有些令人担忧。。。”听到这话时才想到这孩子自己身边竟然没有带一只宝可梦，却还是跟着我一起来到了这个危险的地方。不知道她在关东的时候，黄金喷雾会不会不够用。另一个印象深刻的地方是当我们从以太基地出发的那天早上，她换下了一直带着的白色遮阳帽，扎起了马尾辫，那个样子真让人向往呢。

当然这个游戏并不只有剧情，反而来说，剧情大多情况下是个新人入门用的，而对于核心玩家而言，培育和对战才是这个游戏的乐趣所在。我用了差不多20小时结束了二周目，在莉莉艾不在的这段时间里沉迷孵蛋，不能自拔。先是服了20个伊布，开了战斗树的个体值查看，然后用五十多的连锁刷了一只4v百变怪，然后又用奇迹交换来的3v小火龙孵了一盒子蛋，得到了一个5v内敛小火龙，因为我的是y喷而没有满的那一项是攻击，所以5v和6v几乎没有差别，算得上是完美，开心得不行了。不过训练家的道路还刚刚开始，什么时候才能成为聚聚呢？等我有空再说吧。。。

跳表(skip list)

发表于 2017-01-06 | 分类于 data structure ， search |

昨天看redis实现时，发现搜索用到了跳表。以前上数据结构课的时候就学到过跳表，但是当时没有细讲。这两天看了跳表的原理并且用python进行了实现。
感觉跳表这种数据结构既优雅又暴力。

教程：数据映射–跳表(skiplist)
上面的blog里的代码有点问题

《Skip Lists : A Probabilistic Alternative to balance tree》
具体看论文。
二叉搜索树在元素以随机顺序插入时有很好的效果，但是当元素以特定顺序插入时（敌手策略），其性能会急剧下降。对此，如果能随机打乱输入序列的话，二叉搜索树就能对任何输入序列有较好的性能，但是在实际应用中，输入的序列是及时产生的，也就是说不能实现随机的重排序，所以上述方法不能用了。
所以就产生了平衡二叉树（如：AVL树、红黑树），通过实时调整二叉树的结构来保证搜索效率。
而跳表（skip list）也是一种可选的代替方案。跳表不需要实时调整结构仍能保证搜索效率，因为它引入了随机数和概率。通过随机数生成器，保证虽然最坏情况依然很坏，但是没有特定的输入序列会达到这个最坏情况（类似quicksort中随机选取pivot element）。用论文里的说法就是：Skip lists have balance properties similar to that of search trees built by random insertions, yet do not require insertions to be random.

论文里还对比了一下平衡树（AVL trees，self-adjusting trees（这个没听说过。。。））
三者在性能上有相同的bound。对它们的取舍主要有以下因素：实现难度、constant factors、type of bound(amortized, probabilistic or worst-case)、performance on a non-uniform distribution of queries.

1.实现难度上很显然skip list更简单
2.constant factors
constant factors对算法的实际应用非常重要，特别是对sub-linear算法而言。
比如，A和B都是O(logn)的算法，B比A快一倍，那么A处理大小为n的输入时，B可以处理n^2的输入。（之前从来没想过orz）

  • a)算法本身内在的复杂度会限定对它的任何实现的lower bound。self-adjusting trees在搜索过程中会持续重排，加大了在实现中消耗。而skip list相较而言内在的复杂度比较低。
  • b）如果算法太过复杂，那么对算法实现的优化也较难进行。

3.type of bound
4.performance on a non-uniform distribution of queries
在高度倾斜的输入分布下，self-adjusting trees性能比skip list好。
但在这种条件下有更好的选择：《Slow optimally balanced search strategies vs. cached fast uniformly balanced search strategies》

我的游戏日志(1): Ghost Trick(幽灵诡计)

发表于 2017-01-01 | 分类于 ACG ， Game |

作为一个bangumi评分8.9分，游戏区排名第14位的游戏，其品质绝对是极佳的，除了能抓住一批核心玩家之外，也能够吸引到其他普通玩家。
第一次看到这个游戏就是在逛bangumi的游戏区时，当时就标记了个“想玩”。但是直到现在才真正通关了这游戏，主要还是因为它是在nds平台的，准备nds模拟器太麻烦了。不过貌似也有ios平台的，这就不太清楚了。
总之，趁着元旦这两天，从昨天下午一直推到晚上，再加上今天下午，通关完毕。整体而言的话，这个游戏无论从剧情、悬疑、演出、还是游戏性上都是极其优秀的。特别是挖坑、填坑水平一流，前面的埋下的伏笔到最后基本都有很好的解释。而游戏中使用的trick更是它的灵魂所在。

剧情简介：
一个灵魂从昏迷中醒来，发现自己已经被杀死，而自己又失去了之前的全部记忆。而与此同时是，自己尸体旁的不知名女孩正受到杀手的追杀。玩家要操纵这个灵魂，使用灵魂特有的力量“死者之力”，挽救女孩的生命，同时追查自己死亡的原因。时限只有一夜————一旦太阳升起，死者的灵魂就会消散。。。

以下涉及剧透














一开始看到题目《ghost trick》，我就开始猜故事里用的到底是什么trick。然后我脑中浮现出的E17中的两个核心trick(具体什么就不说了)。随着剧情的推进，这两个猜测都被否定了。不过必须得说，如果这游戏里使用了以上任意一个trick，那也太没有新意了。。。
这个故事最大的悬疑有二：一是主角西塞尔的真实身份到底是什么？死因是什么？二是杀手为什么要杀死凛音，以及从中抽出的一系列人物链。
随着故事的深入，我们会发现挖出来的坑越来越多：台灯库纽利是什么来头？十年前发生了什么？为什么乔多的画上是西塞尔？为什么西塞尔要和外国势力交易？交易的内容如何影响国家的命运？导致乔多入狱的5年前的真相是什么？垃圾场地下室的机关布置会和5年前的事故一样？“死者之力”似乎不是西塞尔独有的。
到了故事过半，真相才渐渐揭露。具体就不说了。
比较印象深刻的一个是西塞尔和尤弥尔不是同一个人，准确的说是香蕉头的尸体不是西塞尔。这里使用的是一个玩家先入为主的trick，因为一开场就看到了仅此一个的尸体，就下意识地认为是自己的尸体。这里其实是有伏笔的，因为剧情前期也有那个谁的灵魂一开始把自己当成其他人了。这个trick我是直到看到垃圾场中尤弥尔的尸体复活才看出来的。另一个是西塞尔本体居然是那只黑猫，这个实在是看不出来，毕竟也算是场外因素。不过之前也做了伏笔，就是导弹的灵魂也有人类的思维这一点。所以这两个trick虽然意外，但也可以很容易让人接受。
至于其他剧情方面，基本上很缜密，bug不多，就算有也不是影响关键剧情的那种。
还有值得一说的是最后通过取出陨石的尤弥尔的尸体跳跃到10年前，这个真是十足的逆转，玩的时候不禁发出“还有这手”的感叹。最后陨石碎片击中西塞尔并留在它体内，导致它十年里都没有变化，这点也算是给玩家糖了吧。毕竟对英雄没有一点奖励的话也有点说不过去。不过最后西塞尔感觉就和神一样，和凡人玩游戏的态度，感觉有点不爽呢。。

###分类和标签
只有文章支持分类和标签，您可以在 Front-matter 中设置。在其他系统中，分类和标签听起来很接近，但是在 Hexo 中两者有着明显的差别：分类具有顺序性和层次性，也就是说 Foo, Bar 不等于 Bar, Foo；而标签没有顺序和层次。

  • December

  • PS3
  • Games

1. 文章资源文件夹

对于那些想要更有规律地提供图片和其他资源以及想要将他们的资源分布在各个文章上的人来说，Hexo也提供了更组织化的方式来管理资源。这个稍微有些复杂但是管理资源非常方便的功能可以通过将 config.yml 文件中的 post_asset_folder 选项设为 true 来打开。

当资源文件管理功能打开后，Hexo将会在你每一次通过 hexo new [layout] 命令创建新文章时自动创建一个文件夹。这个资源文件夹将会有与这个 markdown 文件一样的名字。将所有与你的文章有关的资源放在这个关联文件夹中之后，你可以通过相对路径来引用它们，这样你就得到了一个更简单而且方便得多的工作流。

2. 相对路径引用的标签插件

3. 列表

Markdown 支持有序列表和无序列表。

无序列表使用星号、加号或是减号作为列表标记

  • Red
  • Green
  • Blue
    等同于：
  • Red
  • Green
  • Blue
    也等同于：
  • Red
  • Green
  • Blue

有序列表则使用数字接着一个英文句点：

  1. Bird
  2. McHale
  3. Parish
    很重要的一点是，你在列表标记上使用的数字并不会影响输出的 HTML 结果，上面的列表所产生的 HTML 标记为：

  1. Bird

  2. McHale

  3. Parish


    4. 如果你的列表标记写成：

    1. Bird
    2. McHale

    3. Parish
      或甚至是：

    4. Bird

    5. McHale
    6. Parish
      你都会得到完全相同的 HTML 输出。

    • A list item with a blockquote:

      inside a list item.

      换句话说，也就是在行首出现数字-句点-空白，要避免这样的状况，你可以在句点前面加上反斜杠。

    1986. What a great season.

    2. 强调

    Markdown 使用星号（）和底线（_）作为标记强调字词的符号，被 或 包围的字词会被转成用 标签包围，用两个 * 或 包起来的话，则会被转成 ，例如：

    single asterisks

    single underscores

    double asterisks

    double underscores

    你可以随便用你喜欢的样式，唯一的限制是，你用什么符号开启标签，就要用什么符号结束。
    unfriggingbelievable
    但是如果你的 * 和 _ 两边都有空白的话，它们就只会被当成普通的符号。

    如果要在文字前后直接插入普通的星号或底线，你可以用反斜线：

    *this text is surrounded by literal asterisks*

    3. 代码

    如果要标记一小段行内代码，你可以用反引号把它包起来（`），例如：

    Use the printf() function.

    如果要在代码区段内插入反引号，你可以用多个反引号来开启和结束代码区段：

    There is a literal backtick (`) here.

    代码区段的起始和结束端都可以放入一个空白，起始端后面一个，结束端前面一个，这样你就可以在区段的一开始就插入反引号：

    A single backtick in a code span: `

    A backtick-delimited string in a code span: `foo`

    其它

    1. 自动链接

    Markdown 支持以比较简短的自动链接形式来处理网址和电子邮件信箱，只要是用尖括号包起来， Markdown 就会自动把它转成链接。一般网址的链接文字就和链接地址一样，例如：

    http://example.com/

    邮址的自动链接也很类似，只是 Markdown 会先做一个编码转换的过程，把文字字符转成 16 进位码的 HTML 实体，这样的格式可以糊弄一些不好的邮址收集机器人，例如：

    address@example.com

    2. 反斜杠

    Markdown 可以利用反斜杠来插入一些在语法中有其它意义的符号，例如：如果你想要用星号加在文字旁边的方式来做出强调效果（但不用 标签），你可以在星号的前面加上反斜杠：

    *literal asterisks*

    Markdown 支持以下这些符号前面加上反斜杠来帮助插入普通的符号：

    \ 反斜线
    ` 反引号
    * 星号
    _ 底线
    {} 花括号
    [] 方括号
    () 括弧
    # 井字号
    + 加号
    - 减号
    . 英文句点
    ! 惊叹号
